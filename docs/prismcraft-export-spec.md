# PrismCraft → Voxel Editor: GLB Export Spec (v1)

Audience: the Claude Code instance maintaining PrismCraft's block pipeline.
Consumer: the Godot 4 voxel map editor / game in `ActionAdventureVoxel`
(loads `blocks/*.glb` at runtime, renders via MultiMesh instancing).

## Why this spec exists

The current GLB exports bake each block's 32×32 texture **as geometry** —
one quad per texel. Measured impact in the consumer:

| shape (examples) | triangles today | triangles needed |
|---|---|---|
| `grass.cube` | 8,624 | **12** |
| `dirt.cube` | 9,132 | **12** |
| `stone.ramp` | 4,020 | ~**24** |
| `oak.octagon` | 3,536 | ~**36** |
| `slate.gable` | 5,210 | ~**24** |

A map renders tens of thousands of block instances; at ~9k triangles per
cube the ground alone exceeded 200M triangles/frame and performance
collapsed. The engine now works around this for `.cube` blocks only (it
re-extracts the texture from the texel-quads at load and substitutes a
12-triangle cube). All other shapes still pay the full cost. **The fix
belongs at export time, in PrismCraft.**

## The core requirement

> Export low-poly UV-mapped geometry with the 32×32 texture embedded as an
> actual texture. Never subdivide faces to encode texels.

Per-face detail must come from the sampled texture, not vertex density.

## GLB requirements

### Geometry
1. **Unit cell**: the block occupies x ∈ [−0.5, 0.5], y ∈ [0, 1],
   z ∈ [−0.5, 0.5]. Origin = bottom-center. Y-up. (Unchanged from today.)
2. **Minimal triangulation**: each planar face is a single quad (2 tris)
   or the minimal set for the shape. Budget guide: `cube` 12 tris,
   `slab-*` 12, `ramp` ≤ 24, `gable` ≤ 24, `stairs-2` ≤ 44, `stairs-4` ≤ 84,
   `octagon`/`octagon-half` ≤ 80, `opening`/`diagwall`/columns ≤ 200.
   Hard ceiling for any block: **500 triangles**.
3. **Indexed** triangles, one mesh, one primitive per material (ideally
   exactly one material). Single node at the origin — no transforms, no
   hierarchy, no animations, no skins.
4. **Normals** included, flat per face (no smoothing across face edges).
5. **Orientation conventions** (unchanged — the map generator depends on
   these native orientations; rotation is applied at placement):
   - `ramp` / `stairs-*`: rise toward **+X** (low edge at −X)
   - `gable`: ridge runs along **Z**, slopes descend toward ±X
   - `opening`: passage pierces along **Z**

### Texturing
6. **UV0 required** on every vertex. Texel edges must land exactly on UV
   grid lines so nearest-filtering reproduces the crisp 32×32 look.
7. **Embed the texture** in the GLB as the material's
   `baseColorTexture` (PNG inside the binary chunk is fine). Use the
   existing per-shape texture sheets (`<material-variant>_<shape>_<WxH>.png`)
   as-is; the UV layout may follow the sheet's own atlas layout — the
   consumer only cares that UVs and texture agree.
8. **Sampler**: `magFilter = NEAREST` (9728), `minFilter = NEAREST` or
   `NEAREST_MIPMAP_NEAREST`. Wrap = CLAMP_TO_EDGE.
9. **Material**: single material named `voxel`; `metallicFactor = 0`,
   `roughnessFactor = 1`, `doubleSided = false`, `baseColorFactor = [1,1,1,1]`.
   (The consumer replaces the material with its own cel shader but reads
   the baseColorTexture from it.)
10. **Alpha**: only `leaves` (and future cutout materials) may carry
    alpha; use `alphaMode = MASK`, `alphaCutoff = 0.5`. Everything else
    `OPAQUE`.
11. **Vertex colors (COLOR_0)**: drop them. If the pipeline keeps them
    for its own tooling they will be ignored once a baseColorTexture is
    present.

### Naming & placement (unchanged)
12. Files stay `<material-variant>.<shape>.glb` in `blocks/` — the
    consumer discovers blocks by filename and treats the name as the
    block ID, per the PrismCraft naming convention v3.6.

## Consumer behavior (so you can verify against it)

- Detection: if a `.cube` GLB's first primitive has a `baseColorTexture`,
  the engine uses the mesh as-is; if it only has COLOR_0 texel-geometry,
  the engine falls back to its extraction workaround (slower load, cubes
  only). Non-cube shapes are always used as-is — that's why their
  re-export matters most.
- The engine assigns its own cel-shaded material at load, sampling the
  embedded texture with nearest filtering; roughness/metallic and any
  PBR extensions in the GLB are ignored.
- Collision is generated by the engine (box for `cube`/`opening`, ramp
  wedge for `stairs-*`, convex hull otherwise) — nothing needed from
  PrismCraft.

## Validation checklist (per exported GLB)

- [ ] bounds exactly x,z ∈ [−0.5, 0.5], y ∈ [0, 1] (± 0.001)
- [ ] triangle count within the shape budget (≤ 500 hard ceiling)
- [ ] exactly one mesh, one node, no transform/animation/skin
- [ ] UV0 present; baseColorTexture embedded; sampler NEAREST
- [ ] material `voxel`, metallic 0, roughness 1, correct alphaMode
- [ ] ramp/stairs rise toward +X; gable ridge along Z; opening pierces Z
- [ ] renders identically to the old texel-geometry version at 1:1 zoom

## Migration note

Re-export **all 115 blocks**. Priority order if staged: (1) `ramp`,
`gable`, `stairs-*`, `octagon*` — the consumer has no workaround for
these; (2) `cube` — currently worked around but load time and memory
still suffer; (3) the rest. The consumer needs no code changes to accept
compliant files, and its `.cube` workaround will be retired once the
re-export lands.
